# SKILL: Feedback Tone & Learning Encouragement

## When to Use This Skill
When the feedback tone is too blunt, or a student might feel discouraged.
Or when there is a request like "this team is beginners, please be gentle."

---

## Tone Guidelines

### Expressions to Avoid
| Avoid | Use Instead |
|---|---|
| "This is wrong." | "What do you think could happen in this part?" |
| "Why did you do it this way?" | "Was there a reason you made this choice?" |
| "This is bad code." | "What do you think would improve if we refined this part?" |
| "Obviously you should ~" | "The general reason for ~ is ~" |

### Expressions to Keep
- Mention **potential** before problems
- Use suggestion forms like "What if you tried ~?" / "Have you considered ~?"
- Lead with what went well, follow with areas for improvement

---

## Adjusting Tone by Team Level

| Team Level | Feedback Focus |
|---|---|
| Beginner (first time coding) | Emphasize what went well; only one key point |
| Elementary (1–2 semesters in) | Focus on readability and naming |
| Intermediate (has project experience) | Cover structure, patterns, and collaboration practices |

---

## Even Praise Should Include a Reason

| Weak Praise | Strong Praise |
|---|---|
| "Good job!" | "The function names describe their roles precisely, making the code easy to read." |
| "The code looks clean." | "The consistent indentation and spacing make it easy to follow the flow." |
